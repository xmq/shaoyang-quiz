# 软件工程 · 三色回忆卡

> 先答后核对。🔴是定义、原则和必写结论；🔵是分析、建模、测试与计算步骤；⚫是适用条件和常见扣分点。展开解释与案例见“讲义”。

## 一、软件工程与过程

### 回忆卡 1｜软件工程与生命周期

先答：软件工程解决什么问题？生命周期通常包含哪些活动？

- 🔴 软件工程用系统化、规范化、可量化方法开发、运行和维护软件；生命周期含计划、需求、设计、编码、测试、部署和维护。
- 🔵 软件危机题按“复杂度增长 → 需求/估算/管理不足 → 延期、超支、低质量、难维护”组织。
- ⚫ 生命周期活动可迭代，不等于所有项目必须严格顺序执行。

### 回忆卡 2｜过程模型选择

先答：瀑布、原型、增量、螺旋、敏捷各适合什么项目？

- 🔴 瀑布适合需求稳定和强合规；原型澄清需求；增量分批交付；螺旋突出风险分析；敏捷用短迭代持续反馈。
- 🔵 按需求稳定性、风险、规模、交付频率、文档合规和团队能力给出“模型+理由+控制措施”。
- ⚫ 敏捷不等于不要文档计划；低质量原型不能无控制直接变产品。

## 二、可行性与需求工程

### 回忆卡 3｜可行性与需求层次

先答：可行性分析看什么？业务、用户、系统、功能与非功能需求怎样区分？

- 🔴 可行性含技术、经济、操作/社会、法律、进度；业务需求定目标，用户需求定任务，系统需求细化功能与质量约束。
- 🔵 非功能需求写成可测指标，如响应时间、可用性、安全级别和兼容范围。
- ⚫ 沉没成本不应影响未来决策；“系统要友好、快速”不可验证，不是合格规格。

### 回忆卡 4｜需求过程、用例与追踪

先答：需求工程怎样形成闭环？一份用例至少写哪些内容？

- 🔴 获取 → 分析协商 → SRS → 验证 → 变更管理；SRS应正确、完整、一致、无歧义、可验证、可追踪。
- 🔵 用例写参与者、目标、前置条件、主成功场景、异常流、后置条件；变更先做影响分析，再更新需求—设计—代码—测试追踪。
- ⚫ 参与者是外部角色；`include`是必用公共行为，`extend`是条件扩展，用例不能写成按钮操作清单。

## 三、结构化分析与设计

### 回忆卡 5｜DFD与数据字典

先答：DFD包含哪些元素？分层时怎样检查平衡和非法数据流？

- 🔴 DFD含外部实体、加工、数据流、数据存储；上下文图只画一个总加工，父子图输入输出必须平衡。
- 🔵 先定系统边界 → 画上下文图 → 逐层分解加工 → 用数据字典定义数据项、结构、存储和加工逻辑。
- ⚫ 外部实体、数据存储之间不能绕过加工直接传数据；加工要避免黑洞、奇迹和灰洞。

### 回忆卡 6｜模块化与体系结构

先答：高内聚低耦合为何重要？常见体系结构怎样选？

- 🔴 模块化依靠抽象和信息隐藏控制复杂度；分层隔离职责，C/S分布服务，MVC分模型/视图/控制，仓库共享数据，管道适合变换流，事件驱动解耦生产与消费。
- 🔵 概要设计按子系统划分 → 职责接口 → 数据控制关系 → 架构选择 → 质量属性与故障检查。
- ⚫ 内聚耦合分类以教材为准；事件解耦会增加追踪和一致性难度，不能只写优点。

## 四、面向对象与UML

### 回忆卡 7｜对象、原则与类关系

先答：封装、继承、多态是什么？依赖、关联、聚合、组合、泛化、实现怎样区分？

- 🔴 对象有状态、行为、标识；封装隐藏实现，继承表达复用与is-a，多态使同一接口有不同实现。组合生命周期依赖最强，泛化是继承，实现是类履行接口。
- 🔵 关系题从使用时长、所有权、生命周期、导航和多重性判断并标在类图上。
- ⚫ 不能只凭“成员变量”判组合；SOLID用于权衡职责与依赖，机械套用会过度设计。

### 回忆卡 8｜UML图选择

先答：用例、类、顺序、活动、状态、部署图分别回答什么问题？

- 🔴 用例图看外部功能，类图看静态结构，顺序图看时间消息，活动图看流程并发，状态图看对象迁移，部署图看软硬件结点。
- 🔵 顺序图识别参与者/边界/控制/实体并用alt、opt、loop；状态迁移写`事件[守卫]/动作`并检查初终态。
- ⚫ 图必须与文字、需求一致；只画符号不解释职责和约束不能形成完整答案。

## 五、设计细化

### 回忆卡 9｜接口、数据、界面与模式

先答：详细设计至少要把哪些契约写清？设计模式题怎样得分？

- 🔴 接口写输入输出、前后置条件、错误、幂等、超时和版本；数据设计考虑模式、索引、事务与一致性；界面强调一致、反馈、容错、可访问。
- 🔵 模式题按“意图 → 结构/角色 → 场景 → 代价”答；流程图、N-S、PAD或伪代码都应使用可验证的顺序、选择、循环。
- ⚫ 单例会带来隐藏依赖，观察者要处理通知顺序和取消订阅；模式不是越多越好。

## 六、软件测试

### 回忆卡 10｜测试层次与方法

先答：单元、集成、系统、验收、回归测试分别验证什么？静态、动态、黑盒、白盒怎样区分？

- 🔴 测试用于发现缺陷；单元测模块，集成测接口，系统测完整需求，验收由用户确认业务，修改后做回归。静态不执行程序，动态要执行。
- 🔵 从需求建立测试目标和层次，再组合黑盒规格测试与白盒结构测试。
- ⚫ 测试不能证明绝对无错；自动化覆盖高也不能替代探索性和验收测试。

### 回忆卡 11｜黑盒用例设计

先答：等价类、边界值、判定表怎样产生完整测试用例？

- 🔴 等价类划有效/无效类；边界值取上下界及邻近点；判定表列条件、组合规则和动作。
- 🔵 写测试目标 → 划分输入 → 列输入/执行条件/预期结果 → 标覆盖的类、边界或规则。
- ⚫ 多个无效输入通常分开测以便定位；只有输入没有预期结果不算完整测试用例。

### 回忆卡 12｜白盒覆盖与环形复杂度

先答：覆盖强度怎样比较？怎样求McCabe复杂度并设计基本路径？

- 🔴 语句覆盖弱于分支覆盖；条件与分支覆盖互不包含，判定—条件二者均满足。`V(G)=E−N+2=判定结点数+1=区域数`。
- 🔵 画控制流图并编号 → 求V(G) → 列V(G)条基本独立路径 → 为每条写输入和期望 → 核对覆盖目标。
- ⚫ 循环使完整路径可能无限；复合条件按实际控制流图判定结点口径。

### 回忆卡 13｜集成策略

先答：自顶向下、自底向上、大爆炸和三明治集成各需什么测试替身？

- 🔴 自顶向下用桩模块，早测主控；自底向上用驱动模块，早测基础服务；大爆炸定位故障困难；三明治混合两者。
- 🔵 根据关键风险和模块可用顺序制定集成批次，并在每批执行接口与回归测试。
- ⚫ 桩模拟被调用下层，驱动模拟调用上层，二者不能写反。

## 七、质量、维护与配置

### 回忆卡 14｜质量、维护与可靠性

先答：四类维护怎样区分？可靠性、可用性怎样表述？

- 🔴 纠错修缺陷，适应应环境，完善增功能/性能，预防性重构降未来风险；可用性约`MTBF/(MTBF+MTTR)`。
- 🔵 把质量属性写成可测目标，并从模块化、文档、复杂度、测试和架构说明可维护性。
- ⚫ 重构不改变外部可观察行为且需测试保护；可靠性与可用性不是同一概念。

### 回忆卡 15｜配置与变更管理

先答：配置管理包含哪些活动？基线、分支、标签分别做什么？

- 🔴 配置管理含配置项识别、版本控制、变更控制、状态记录、审计、构建发布；基线经评审批准，只能按流程修改。
- 🔵 变更按申请 → 影响分析 → 审批 → 实施验证 → 更新基线和追踪记录。
- ⚫ 分支支持并行，标签标发布点；语义化版本兼容规则仍以项目约定为准。

## 八、估算、进度与风险

### 回忆卡 16｜规模、COCOMO与PERT

先答：LOC、功能点怎样比较？COCOMO和PERT公式是什么？

- 🔴 LOC受语言影响且早期难估；功能点较早且相对语言无关。COCOMO：`E=a(KLOC)^b，D=cE^d`；PERT：`t=(a+4m+b)/6`，方差`((b−a)/6)²`。
- 🔵 先写模型系数与独立假设，再算活动期望/方差并沿路径求和。
- ⚫ 增加人员不能任意缩短工期；估算要写假设、区间并持续校准。

### 回忆卡 17｜关键路径与挣值

先答：怎样求关键路径、总时差？PV、EV、AC和四个偏差指标怎样解释？

- 🔴 关键路径总工期最长且活动总时差0；`CV=EV−AC，SV=EV−PV，CPI=EV/AC，SPI=EV/PV`，正偏差和指数大于1通常有利。
- 🔵 CPM正推ES/EF取max → 逆推LF/LS取min → `TF=LS−ES` → 列TF=0路径和工期。
- ⚫ 缩短非关键活动不一定缩短项目；多条关键路径时要同时检查。

### 回忆卡 18｜风险与综合大题

先答：风险怎样管理？过程模型、需求建模、测试设计类大题怎样形成证据链？

- 🔴 风险暴露≈概率×损失；流程为识别、分析排序、应对、监控，应对含规避、减轻、转移、接受。
- 🔵 大题统一写“项目/系统边界与目标 → 方法选择 → 图表/计算或用例 → 一致性与覆盖检查 → 风险、代价和改进”。
- ⚫ 已发生的是问题而非风险；图必须有文字解释，结论必须能追踪到需求、数据或计算。

## 九、敏捷与DevOps

### 回忆卡 19｜Scrum、XP与CI/CD

先答：Scrum角色与事件、XP实践、持续集成/交付/部署怎样区分？

- 🔴 产品负责人管价值与待办，Scrum Master促成过程，开发者交付增量；XP含TDD、结对、持续集成、重构；持续交付保证可发布，持续部署自动上生产。
- 🔵 用“短反馈周期 + 自动构建测试 + 质量门槛DoD + 可回滚发布”解释工程闭环。
- ⚫ 敏捷不等于不控范围，CI/CD不等于取消安全审批、验收、监控和回滚。

## 十、大题书写模板

### 回忆卡 20｜分析与设计题证据链

先答：过程模型选择、需求建模、测试设计和设计评价怎样写成可评分答案？

- 🔴 完整答案必须给出项目/系统特征、所选方法、图表或用例、检查结果及风险代价。
- 🔵 按“边界与目标 → 方法和理由 → 图/计算/测试用例 → 一致性或覆盖检查 → 改进方案”展开。
- ⚫ 只有模型名称、输入数据或空泛优缺点都不完整；图中元素必须能追踪到文字和需求。
