# 数据库技术：三色笔记

三色按考频和出题稳定性划分：红色优先背熟，蓝色用于补齐得分点，黑色用于防漏和边界判断。

## 🔴 高频考点

- 🔴 **数据库基本概念**：数据库、DBMS、数据库系统分别指数据集合、管理软件和完整系统。 常见考法：判断三者区别。 易错边界：数据库不等于数据库软件。
- 🔴 **关系模型**：二维表、行、列、主键、外键构成关系数据库基础。 常见考法：判断主键、外键、实体完整性。 易错边界：主键不能重复且不能为空。
- 🔴 **SQL 查询**：`SELECT-FROM-WHERE` 是最常用查询结构。 常见考法：写简单查询、条件、排序、分组。 易错边界：`WHERE` 先筛行，`HAVING` 筛分组。
- 🔴 **E-R 模型**：实体、属性、联系用于概念设计。 常见考法：按场景画实体关系，判断 1:1、1:n、m:n。 易错边界：多对多通常要转成中间表。
- 🔴 **范式**：范式用于减少数据冗余和更新异常。 常见考法：判断部分依赖、传递依赖。 易错边界：范式越高不一定查询越快。
- 🔴 **事务 ACID**：原子性、一致性、隔离性、持久性保证可靠操作。 常见考法：解释转账、订单扣库存为什么用事务。 易错边界：事务提交后才强调持久保存。

## 🔵 中频考点

- 🔵 **索引**：索引提高查询速度，但会增加维护成本。 常见考法：判断何时建索引。 易错边界：索引不是越多越好。
- 🔵 **并发控制**：锁和隔离级别用于减少脏读、不可重复读、幻读。 常见考法：判断并发异常。 易错边界：隔离越高并发性能通常越低。
- 🔵 **视图**：视图是基于查询的虚表。 常见考法：判断视图用途。 易错边界：视图不一定真正存储数据。
- 🔵 **数据库设计流程**：需求分析、概念设计、逻辑设计、物理设计。 常见考法：排序设计阶段。 易错边界：E-R 图属于概念设计。
- 🔵 **备份与恢复**：备份、日志和恢复机制保证故障后数据可恢复。 常见考法：判断故障处理措施。 易错边界：备份不等于实时高可用。

## ⚫ 低频了解

- ⚫ **NoSQL**：键值、文档、列族、图数据库适合不同非关系场景。 常见考法：判断 Redis、MongoDB 等类型。 易错边界：NoSQL 不是完全不用结构。
- ⚫ **数据仓库**：面向分析，强调主题、集成、稳定和时变。 常见考法：区别 OLTP 与 OLAP。 易错边界：数据仓库通常不用于高频交易写入。
- ⚫ **分布式数据库**：分片、复制提高扩展性和可靠性。 常见考法：概念判断。 易错边界：分布式会带来一致性和延迟问题。

## 考前规则与简答

### 必会规则

- **SQL执行逻辑**：先由 `FROM` 确定数据来源，`WHERE` 筛选行，`GROUP BY` 分组，`HAVING` 筛选组，`SELECT` 形成结果列，最后 `ORDER BY` 排序。书写顺序与概念执行顺序不要混淆。
- **键约束**：主键唯一且非空，用于标识一行；外键引用另一表的候选键或主键，用于维护参照完整性。外键可以重复，是否允许为空由业务和约束决定。
- **事务边界**：事务内操作要么全部成功提交，要么失败回滚。提交前的中间状态不应被其他事务随意看到，提交后结果应持久保存。

### 常见简答题

- **事务ACID分别表示什么？** 原子性保证操作整体完成或整体回滚；一致性保证约束不被破坏；隔离性控制并发事务相互影响；持久性保证提交结果在故障后仍可恢复。
- **索引为什么不是越多越好？** 索引能加快特定查询，但占用空间，并使插入、删除、更新需要同步维护；应按高频查询条件、选择性和实际执行计划建立。
