# 数据库技术 · 三色回忆卡

> 使用方法：先口述或默写“先答”，再核对答案。🔴只记最小结论；🔵只记 SQL、计算或设计步骤；⚫只记条件与反例。代码均放在问题之后，避免提前暴露答案。

## 一、系统、建模与关系模型

### 回忆卡 1｜数据库系统与三级模式

先答：DB、DBMS、DBS 怎样区分？三级模式和两级映像分别保障什么？

- 🔴 DB 是数据集合，DBMS 管理数据，DBS 还包括应用、人员和环境；外模式面向用户，模式描述全局逻辑，内模式描述物理存储。
- 🔵 由外到内画“外模式—外/模映像—模式—模/内映像—内模式”，前者支持逻辑独立性，后者支持物理独立性。
- ⚫ 一个库通常一个模式和内模式、多个外模式；独立性是尽量少改应用，不是绝对不改。

### 回忆卡 2｜E-R 建模

先答：怎样从业务文字识别实体、属性、联系、基数和弱实体？

- 🔴 可独立存在且有标识者建实体，描述信息建属性，业务动词建联系；联系有 1:1、1:n、m:n，弱实体依赖所有者并有部分码。
- 🔵 圈名词与标识 → 圈动词 → 标基数/参与 → 检查多值属性和冗余联系 → 用实例验证。
- ⚫ “出现一个名词”不必然建实体；基数必须按业务约束而非表面语句猜测。

### 回忆卡 3｜E-R 转关系模式

先答：1:1、1:n、m:n、多值属性和弱实体分别怎样转表？

- 🔴 1:n 把 1 端码放到 n 端作外码；m:n 单独建联系表；多值属性单独建表；弱实体表含所有者码；1:1 可择一方放外码，常选全参与方。
- 🔵 每张表写属性、主码、外码，再检查联系属性和业务唯一性。
- ⚫ 把外码放到 1:n 的 1 端会丢失多值；联系表主码是否只由两端码组成取决于题意。

### 回忆卡 4｜键、完整性与 NULL

先答：超码、候选码、主码、主属性、外码和三类完整性是什么？NULL 怎样判断？

- 🔴 候选码是最小超码，选中者为主码，候选码中的属性为主属性；外码引用候选/主码。完整性分实体、参照、用户定义。
- 🔵 涉及 NULL 用 `IS NULL/IS NOT NULL`；`COUNT(列)` 忽略 NULL，`COUNT(*)` 统计行。
- ⚫ NULL 不等于 0 或空串；普通比较产生 UNKNOWN；外码可空须受列约束允许。

## 二、关系代数与 SQL

### 回忆卡 5｜关系代数

先答：选择、投影、连接、除法分别表达什么？“全部”语义优先想到什么？

- 🔴 选择筛行，投影取列并按集合语义去重，连接按条件配对，除法表达“与指定集合每个元素都有关系”。
- 🔵 先定输出列 → 早选择缩行 → 保留连接属性地早投影 → 按键连接 → 最后投影；“全部”考虑除法或双重不存在。
- ⚫ 同名属性不一定同义，不能无脑自然连接；漏连接条件会产生笛卡尔积。

### 回忆卡 6｜SQL 查询骨架

先答：默写含连接、筛选、分组、组筛选和排序的查询骨架，并说出逻辑执行顺序。

- 🔴 逻辑顺序是 `FROM/JOIN → WHERE → GROUP BY → HAVING → SELECT → DISTINCT → ORDER BY`。
- 🔵 默写：`SELECT ... FROM A JOIN B ON ... WHERE ... GROUP BY ... HAVING ... ORDER BY ...;`
- ⚫ SELECT 中非聚合列原则上必须在 GROUP BY 中；产品的 LIMIT/TOP/FETCH 语法不同。

### 回忆卡 7｜连接与外连接

先答：INNER、LEFT、RIGHT、FULL JOIN 保留哪些行？右表过滤放 ON 和 WHERE 有什么不同？

- 🔴 INNER 只留匹配；LEFT/RIGHT 保留指定一侧；FULL 保留两侧并以 NULL 补未匹配。
- 🔵 先写主外键连接条件，再把“是否仍保留无匹配行”作为判断过滤位置的标准。
- ⚫ LEFT JOIN 后把右表条件写 WHERE，常会过滤补出的 NULL，使结果退化为内连接；FULL 并非所有 DBMS 支持。

### 回忆卡 8｜分组与聚合

先答：WHERE 与 HAVING 分别筛什么？怎样处理分组计数、去重和并列最大值？

- 🔴 WHERE 分组前筛行，HAVING 分组后筛组；`COUNT(DISTINCT x)` 先去重再计数。
- 🔵 写“连接/行筛选 → 分组键 → 聚合 → HAVING”；并列最大可用子查询等于最大值、排名或反连接。
- ⚫ WHERE 不能直接放聚合条件；`COUNT(*)` 与 `COUNT(x)` 在 NULL 下结果可能不同。

### 回忆卡 9｜子查询与“全部”

先答：IN、EXISTS、相关子查询怎样区别？“满足全部目标”怎样用双重 NOT EXISTS 表达？

- 🔴 IN 判断值是否在集合，EXISTS 判断是否存在行；相关子查询引用外层当前行。“全部”即不存在一个目标，使该对象与目标的关系不存在。
- 🔵 默写语义骨架：`WHERE NOT EXISTS(目标 t WHERE NOT EXISTS(关系 r WHERE r.对象=外层 AND r.目标=t))`。
- ⚫ `NOT IN` 遇 NULL 可能整体 UNKNOWN；按计数法写“全部”必须处理重复和空目标集。

### 回忆卡 10｜DDL、约束、视图与索引

先答：怎样在建表时表达主码、外码、唯一、非空和检查约束？视图与索引解决什么？

- 🔴 约束关键字为 PRIMARY KEY、FOREIGN KEY、UNIQUE、NOT NULL、CHECK；视图提供逻辑隔离，索引用空间和维护代价换查询速度。
- 🔵 设计题先写列和类型，再写主/外码与业务约束，最后按筛选、连接、排序列考虑索引。
- ⚫ 复杂连接/聚合视图常不可更新；B+ 树适合等值和范围，哈希偏等值；索引会拖慢写操作。

## 三、函数依赖与规范化

### 回忆卡 11｜函数依赖与 Armstrong 公理

先答：完全、部分、传递依赖怎样区分？Armstrong 三条公理是什么？

- 🔴 X→Y 表示 X 相同则 Y 相同；去掉 X 任一属性均不成立是完全依赖，可去掉是真部分依赖；经中间属性决定为传递依赖。公理是自反、增广、传递。
- 🔵 推导时只用依赖关系逐步写合并、分解或伪传递，不用样例表中的偶然相同。
- ⚫ 函数依赖是所有合法实例上的约束，不是某一张当前表的巧合。

### 回忆卡 12｜属性闭包与候选码

先答：怎样求 X⁺并据此找候选码？

- 🔴 X⁺ 从 X 开始，若某依赖左部已包含就加入右部，直到不变；闭包含全部属性则 X 是超码，且无真子集为超码才是候选码。
- 🔵 先把不出现在任何 FD 右部的属性作为必选起点，再补属性求闭包，得到超码后逐个删属性检验最小性。
- ⚫ 属性多不等于候选码；必须列每轮新加入的属性，避免循环依赖误判。

### 回忆卡 13｜最小依赖集

先答：怎样把函数依赖集化为最小依赖集？

- 🔴 右部单属性、左部无多余属性、集合中无冗余依赖即为最小依赖集。
- 🔵 拆右部 → 对每条 FD 逐个删左部属性求闭包 → 暂删整条 FD 求闭包判断冗余 → 展示时可合并同左部。
- ⚫ 检查依赖冗余必须暂时删除该依赖；左部最小化和整条删除不是同一步。

### 回忆卡 14｜1NF、2NF、3NF、BCNF

先答：四级范式逐级消除什么问题？怎样用函数依赖判定？

- 🔴 1NF 属性原子；2NF 消除非主属性对候选码的部分依赖；3NF 要求每个非平凡 X→A 中 X 为超码或 A 为主属性；BCNF 要求决定因素 X 都是超码。
- 🔵 先求候选码和主属性，再逐条检查 FD，指出第一条违反条件的依赖及最高范式。
- ⚫ BCNF 严于 3NF，但分解可能不保持全部依赖；只有单属性候选码时不会出现对该码的部分依赖。

### 回忆卡 15｜无损连接与保持依赖

先答：R 分成 R1、R2 时怎样判无损？无损与保持依赖分别保证什么？

- 🔴 若交集能函数决定 R1 或 R2，则二元分解无损；无损防伪元组/信息丢失，保持依赖让约束可在子模式内检查。
- 🔵 求 `R1∩R2` 的闭包，看是否覆盖任一子模式；多子模式按题目用追赶表。
- ⚫ 无损不自动保持依赖，保持依赖也不自动无损，必须分别验证。

### 回忆卡 16｜3NF 综合法

先答：怎样由最小依赖集构造无损且保持依赖的 3NF 分解？

- 🔴 每个 X→A 建 XA 模式，合并同左部，删除被包含模式；若没有模式含候选码，再补一个候选码模式。
- 🔵 顺序是“最小依赖集 → 候选码 → 生成模式 → 去包含 → 验码 → 验覆盖与依赖”。
- ⚫ 直接按原 FD 分表可能冗余；范式越高不代表查询一定更快。

## 四、事务、并发与恢复

### 回忆卡 17｜ACID 与并发异常

先答：ACID 四性质和脏读、不可重复读、幻读、丢失修改分别是什么？

- 🔴 ACID 是原子、一致、隔离、持久；脏读读未提交值，不可重复读是同一行前后不同，幻读是谓词结果行集合变化，丢失修改是写覆盖。
- 🔵 案例先画事务时序，再标读写对象、提交点和哪次读/写被影响。
- ⚫ 一致性是业务约束整体成立，不等同“两个副本值总相同”。

### 回忆卡 18｜优先图

先答：怎样用优先图判定调度是否冲突可串行化？

- 🔴 不同事务对同一数据且至少一个写才冲突；图无环则可串行化，拓扑序给等价串行顺序。
- 🔵 每事务一个点；按先后冲突 Ti→Tj 加边；查环并给拓扑序。
- ⚫ 读读不冲突；只看相邻操作会漏边；可串行化不等于原调度是串行的。

### 回忆卡 19｜S/X 锁与 2PL

先答：S 锁、X 锁怎样相容？2PL 与严格 2PL 保证什么、不能保证什么？

- 🔴 S-S 相容，含 X 的不同事务请求不相容；2PL 先增长后收缩，保证冲突可串行化；严格 2PL 把 X 锁持有到提交/回滚。
- 🔵 封锁题逐操作检查锁表和等待边，必要时画等待图判死锁。
- ⚫ 2PL 仍可能死锁；隔离级别的幻读处理依 DBMS 和锁/MVCC 实现。

### 回忆卡 20｜WAL 与故障恢复

先答：WAL 的两条先后约束是什么？怎样由日志确定 UNDO/REDO？

- 🔴 数据页落盘前相关日志先落盘；提交成功前提交日志落盘。立即更新通常 UNDO 未提交并 REDO 已提交，延迟更新通常只 REDO。
- 🔵 从检查点扫描日志分出已提交/未提交 → REDO 正序 → UNDO 逆序 → 写恢复后值。
- ⚫ 具体集合服从题设日志协议；不能仅凭最后一条更新日志判断事务已提交。

### 回忆卡 21｜数据库设计题收口

先答：数据库设计六阶段是什么？综合 SQL/模式题怎样检查答案？

- 🔴 需求 → 概念 → 逻辑 → 物理 → 实施 → 运行维护。
- 🔵 检查“主外码/约束 → 连接条件 → 分组与重复 → NULL → 并列结果 → 无损与依赖 → 必要索引”。
- ⚫ 先保证语义正确再谈性能；反规范化需要额外的一致性维护机制。
