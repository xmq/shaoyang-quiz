# 数据库技术讲义

## 一、数据库系统基础
<!-- exam: A | 单选、判断 -->

### 1. DB、DBMS 与 DBS

- 数据库 DB：长期存储、有组织、可共享的数据集合。
- 数据库管理系统 DBMS：定义、组织、存取、控制和维护数据库的软件。
- 数据库系统 DBS：数据库、DBMS、应用程序、人员和运行环境的整体。

数据库技术的主要特点是数据结构化、共享性高、冗余较低、独立性较强，并由 DBMS 统一控制安全、完整性、并发和恢复。

### 2. 数据模型

数据模型包含数据结构、数据操作和完整性约束三个要素。

常见模型：

- 层次模型：树形结构，一个结点通常只有一个双亲。
- 网状模型：图状结构，可表达多对多关系，但结构较复杂。
- 关系模型：用二维表表示数据，概念简单、使用广泛。
- 面向对象模型：以对象、类和继承等组织复杂数据。

### 3. 三级模式与两级映像

- 外模式：面向具体用户或应用的局部视图。
- 模式：数据库的全局逻辑结构。
- 内模式：数据的物理存储结构。

外模式/模式映像支持逻辑数据独立性，模式/内模式映像支持物理数据独立性。一个数据库通常只有一个模式和一个内模式，可以有多个外模式。

## 二、概念模型与 E-R 图
<!-- exam: A | 单选、判断、场景题 -->

E-R 模型用实体、属性和联系描述现实世界。

- 实体：可相互区别的客观对象，如“学生”。
- 属性：描述实体的特征，如“姓名”。
- 码：能够唯一标识实体的属性或属性组。
- 联系：实体之间的业务关系。

联系基数有 1:1、1:n、m:n。联系基数由两个方向的最大对应数量共同确定：一个 A 最多对应几个 B，以及一个 B 最多对应几个 A。

### E-R 转关系的基本规则

1. 每个实体通常转换为一张表。
2. 1:n 联系把“1”端主码放入“n”端作为外码。
3. m:n 联系单独建立联系表，通常包含两端主码和联系属性。
4. 多值属性单独建表。
5. 1:1 联系可在适当一方加入外码，通常选择全参与方。

注意：外码放在 1:n 的多端；m:n 不能简单在一方添加单个外码，否则无法保存多个对应关系。

## 三、关系模型与完整性
<!-- exam: A | 单选、判断、多选、场景题 -->

### 1. 基本术语

- 关系：一张符合关系模型要求的二维表。
- 元组：表中的一行。
- 属性：表中的一列。
- 域：属性允许的取值集合。

关系中的元组和属性在理论上没有固定顺序，不能出现完全相同的元组。

### 2. 各类键

- 超码：能够唯一标识元组的属性组，可以含多余属性。
- 候选码：不含多余属性的最小超码。
- 主码：从候选码中选定的主要标识。
- 主属性：出现在任一候选码中的属性。
- 外码：引用本表或其他表候选码/主码的属性。

主码一定是候选码，但候选码不一定被选为主码。一个表可以有多个候选码，通常只选一个主码。

**通俗理解：** 键像识别学生的证件：学号和身份证号都可能各自唯一且不可再删减，因此都可成为候选码；学校从中选学号作为日常主标识，它就是主码。把“学号＋姓名”放在一起当然也能唯一识别学生，但姓名是多余的，所以它是超码而不是候选码。外码则像报名表中填写的班级编号，用来指向另一张班级表中已经存在的记录。这个类比只说明唯一标识和引用关系，现实中的“看起来唯一”必须由数据约束保证，不能仅凭经验认定为键。

### 3. 三类完整性

- 实体完整性：主码值唯一且非空。
- 参照完整性：外码为空或等于被参照表中已存在的键值。
- 用户定义完整性：类型、范围、格式和业务规则。

NULL 表示未知、缺失或不适用，不等于 0，也不等于空字符串。判断 NULL 应使用 `IS NULL` 或 `IS NOT NULL`。

## 四、关系运算基础
<!-- exam: B | 单选、判断、填空 -->

- 选择：从关系中选择满足条件的行。
- 投影：选择指定的列，关系代数按集合语义会去重。
- 连接：按相关属性把多个关系组合起来。
- 并：合并两个并相容关系的元组并去重。
- 差：保留属于第一个关系而不属于第二个关系的元组。
- 交：保留同时属于两个并相容关系的元组。
- 笛卡尔积：任取两表元组组合，漏写连接条件时会产生大量无意义结果。
- 除：用于表达“与给定集合中的所有对象都有关联”，例如查找选修了某一组全部课程的学生。

自然连接按所有同名属性相等进行连接，并只保留一份同名列；同名属性不一定业务含义相同，所以不能看到同名就盲目自然连接。

## 五、SQL 基础
<!-- exam: A | 单选、判断、填空、场景题 -->

### 1. SQL 分类

- 数据定义语言 DDL：`CREATE`、`ALTER`、`DROP`。
- 数据查询语言 DQL：`SELECT`。
- 数据操纵语言 DML：`INSERT`、`UPDATE`、`DELETE`。
- 数据控制语言 DCL：`GRANT`、`REVOKE`。
- 事务控制：`COMMIT`、`ROLLBACK`。

`DELETE` 删除满足条件的行，可配 WHERE；`DROP` 删除数据库对象及其定义；`TRUNCATE` 通常快速清空表数据并保留表结构，具体事务行为依 DBMS。

### 2. 建表与约束

```sql
CREATE TABLE Student (
  id   INT PRIMARY KEY,
  name VARCHAR(30) NOT NULL,
  age  INT CHECK (age >= 0),
  dept_id INT,
  FOREIGN KEY (dept_id) REFERENCES Department(id)
);
```

常用约束：`PRIMARY KEY`、`FOREIGN KEY`、`UNIQUE`、`NOT NULL`、`CHECK`、`DEFAULT`。

### 3. 查询基本结构

```sql
SELECT [DISTINCT] 列或表达式
FROM 表
WHERE 行条件
GROUP BY 分组列
HAVING 组条件
ORDER BY 排序列 [ASC|DESC];
```

逻辑上通常先处理 FROM、WHERE、GROUP BY、HAVING，再得到 SELECT 结果并排序。`WHERE` 在分组前筛选行，`HAVING` 在分组后筛选组。

常用聚合函数：`COUNT`、`SUM`、`AVG`、`MAX`、`MIN`。`COUNT(*)` 统计行，`COUNT(列)` 忽略该列中的 NULL。

下面的查询先保留成年学生，再按院系分组，只显示人数不少于5人的院系，并按人数降序排列：

```sql
SELECT dept_id, COUNT(*) AS student_count
FROM Student
WHERE age >= 18
GROUP BY dept_id
HAVING COUNT(*) >= 5
ORDER BY student_count DESC;
```

这里 `WHERE` 处理分组前的行，`HAVING` 处理分组后的组，二者不能任意互换。

### 4. 连接查询

- `INNER JOIN`：只保留匹配行。
- `LEFT JOIN`：保留左表全部行，右表无匹配时补 NULL。
- `RIGHT JOIN`：保留右表全部行。
- `FULL JOIN`：保留两侧全部行，并非所有 DBMS 支持。

需要保留左表无匹配行时，针对右表的过滤条件通常应放在 `ON` 中；放在 `WHERE` 中可能把补出的 NULL 行过滤掉，使左连接接近内连接。

### 5. 子查询与集合运算

`IN` 判断值是否属于子查询结果集合，`EXISTS` 判断子查询是否至少返回一行。相关子查询会引用外层查询中的列。

`UNION` 合并并去重，`UNION ALL` 保留重复。参与集合运算的查询应有相同列数，对应数据类型应兼容。

`NOT IN` 的子查询结果若含 NULL，可能产生未知结果；反向存在性判断常优先考虑 `NOT EXISTS`。

## 六、视图、索引与设计
<!-- exam: B | 单选、判断、场景题、简答 -->

### 1. 视图

视图是由查询定义的虚表，可简化复杂查询、限制用户看到的数据，并增强逻辑独立性。普通视图通常不实际保存结果；物化视图会保存结果并需要刷新。

含多表连接、聚合或 DISTINCT 的复杂视图通常不能直接更新。

### 2. 索引

索引用额外存储和维护成本换取查询速度。B+ 树索引适合等值和范围查询，哈希索引适合等值查询。经常用于筛选、连接、排序且选择性较高的列更适合建索引。

索引并非越多越好：插入、删除和更新时必须维护索引；对索引列做函数、隐式类型转换或前导通配符，也可能降低索引利用率。

场景选择可先看查询条件：订单号精确等值查询适合唯一索引；按下单时间或金额范围筛选、排序适合 B+ 树类有序索引；性别、是否删除等取值很少的列，普通索引未必有明显收益。联合索引还受列顺序影响，不能看到多个查询列就任意建索引。

索引与约束作用不同：索引主要用于加速访问，唯一约束用于保证不重复。DBMS 可能借助唯一索引实现约束，但“性能结构”和“数据规则”仍是两个层面的概念。

**通俗理解：** 数据库索引像书末按关键词和页码排列的索引，先查较小的有序结构，再定位正文记录，常比从第一页逐行翻到最后更快。B+ 树保持键值有序，因此既适合找一个值，也适合查一段范围；但每次新增、删除或修改相关数据，都要同步维护这本“附加目录”。类比的边界是：有索引不保证每次都更快，返回大部分数据、列选择性很低或查询写法不能利用索引时，优化器仍可能选择全表扫描。

### 3. 数据库设计阶段

需求分析 → 概念结构设计 → 逻辑结构设计 → 物理设计 → 实施 → 运行维护。

概念设计常产出 E-R 图；逻辑设计把概念模型转为关系模式；物理设计选择文件组织、索引和存储参数。

## 七、规范化基础
<!-- exam: B | 单选、判断、场景题 -->

规范化用于减少数据冗余和插入、删除、更新异常。

函数依赖 `X→Y` 表示在一个关系中，X的值确定后，Y的值也随之确定。若复合候选码的一部分就能决定某个非主属性，称为部分依赖；若候选码先决定一个非主属性，该属性又决定另一个非主属性，形成传递依赖。理解这些关系后，才能判断一张表为什么需要拆分。

- 1NF：每个属性值不可再分。
- 2NF：在 1NF 基础上，非主属性完全依赖候选码，消除部分依赖。
- 3NF：在 2NF 基础上，非主属性不传递依赖候选码。
- BCNF：每个决定因素都是候选码或超码，要求通常比 3NF 严格。

例如一张“学生—课程—教师”大表反复保存教师信息，会造成冗余：教师信息变化时需要修改多行，删除最后一条选课记录还可能连带丢失教师信息。通常应把学生、课程、教师及其联系拆成结构清晰的关系表。

**通俗理解：** 规范化像把一本混合通讯录按“谁决定谁”拆开保存。若选课表以“学号＋课程号”唯一确定一条记录，而学生姓名只由学号决定，它就依赖复合码的一部分，属于部分依赖；在以课程号为码的课程表中，若课程号决定教师号、教师号再决定教师电话，教师电话就通过教师号传递依赖课程号。把学生、课程、教师及选课关系分别保存，可以减少同一事实的重复和修改不一致。拆分的依据是函数依赖，不是机械地“一列一张表”；范式提高一致性，却可能增加连接，工程中仍需权衡查询性能。

范式越高不代表查询一定越快。工程中可能为性能进行有限反规范化，但必须采取一致性维护措施。

## 八、事务、并发与恢复
<!-- exam: A | 单选、判断、多选、场景题、简答 -->

### 1. ACID

- 原子性：事务中的操作全部完成或全部不做。
- 一致性：事务使数据库从一个一致状态转到另一个一致状态。
- 隔离性：并发事务之间的影响受到控制。
- 持久性：已提交结果在故障后仍能保存。

**通俗理解：** 一次转账可以把 ACID 看成一个受保护的业务过程：扣款和入账必须一起成功或一起撤销是原子性；转账前后都满足余额、账户等规则是一致性；并发转账不能彼此看到不应暴露的中间状态是隔离性；系统确认提交后，即使随后故障，结果仍应能恢复是持久性。类比的边界是：事务机制不会自动创造全部业务规则，一致性还依赖约束和正确程序；持久性也不表示数据永远不会因介质灾难而丢失，仍需日志、备份和恢复措施。

### 2. 并发异常

- 丢失修改：一个事务的写入覆盖另一个事务的修改。
- 脏读：读到其他事务尚未提交的数据。
- 不可重复读：同一事务两次读取同一行结果不同。
- 幻读：同一条件两次查询，满足条件的行集合发生变化。

共享锁 S 允许多个事务读，排他锁 X 不允许其他事务同时读写该数据。封锁可提高隔离性，但可能产生死锁。

SQL 标准常见隔离级别从低到高为读未提交、读已提交、可重复读和串行化。它们与并发异常的最小对应关系如下：

- 读未提交：可能读到其他事务尚未提交的数据，即脏读。
- 读已提交：避免脏读，但同一行前后两次读取仍可能不同，即不可重复读。
- 可重复读：避免脏读和不可重复读；对幻读的控制方式与 DBMS 的锁和多版本并发实现有关。
- 串行化：并发效果等价于事务串行执行，隔离最强，等待和冲突开销通常也最大。

隔离级别越高，通常能减少更多并发异常，但也可能增加等待和冲突；不同 DBMS 对快照、多版本并发和各级别细节的实现不完全相同。

**通俗理解：** 可以把并发事务想成多人查看和修改同一本库存账：读到别人尚未确认、随后撤销的铅笔记录是脏读；自己前后两次查看同一商品，数量被别人提交修改而不同，是不可重复读；两次按“库存少于 10”查询，别人新提交的商品使结果集合多出一行，是幻读。隔离级别逐步限制这些相互可见的变化，以正确性换取更多等待或实现成本。这个类比只建立异常之间的区别，具体 DBMS 可能用锁或多版本并发控制实现，同名隔离级别对幻读和快照可见性的细节不能脱离产品文档一概而论。

### 3. 订单场景中的机制选择

收银系统把“写订单、扣库存、记付款状态”放在同一事务中，是因为这些操作应全部成功或全部失败，主要体现原子性和一致性。多个收银端并发修改同一商品库存时，需要由锁、隔离级别或数据库提供的原子更新控制并发，避免丢失修改和超卖。

数据库事务保证它所覆盖的数据操作，但不能自动保证网络请求绝不重复。客户端超时重试可能把同一订单再次提交，因此业务层还应使用唯一订单号、唯一约束或幂等键。TCP 的可靠有序传输也不能替代数据库事务：TCP 保证一条连接中的字节流，不理解“订单只能创建一次”的业务含义。

隔离不是越强越好。提高隔离级别通常能减少并发现象，但可能增加等待、冲突和开销；实际系统要在正确性要求与并发性能之间取舍。

### 4. 日志与备份

日志记录数据库更新和事务状态，可用于故障恢复。预写日志 WAL 的基本思想是“先写日志，再把相关数据页写入磁盘”。`COMMIT` 表示提交，`ROLLBACK` 表示回滚。

事务故障通常只影响某个事务，系统故障会使内存状态丢失，介质故障还可能破坏磁盘数据。恢复时可依据日志撤销未提交事务、重做已提交但尚未写回的数据；检查点记录一个已知恢复位置，用于减少故障后需要扫描的日志范围。

备份是独立的数据副本，日志用于重做或撤销操作，两者作用不同。RAID 提高可用性但不能代替备份。

## 九、安全管理
<!-- exam: B | 单选、判断、场景题 -->

身份鉴别确认“你是谁”，授权决定“你能做什么”，审计记录“你做过什么”。`GRANT` 用于授权，`REVOKE` 用于收回权限。

安全管理应遵循最小权限原则，通过用户、角色、视图、授权和审计保护数据。完整性约束保证数据正确性，权限控制保证只有合适主体才能访问，两者不能混为一谈。

常见访问控制包括自主访问控制、强制访问控制和基于角色的访问控制。业务系统通常把权限授予角色，再让用户加入角色，以降低逐个用户授权的维护成本；角色仍应按职责拆分，不能把所有人员都加入高权限角色。

例如只负责查询报表的账号可授予指定视图的 `SELECT` 权限，不必获得基础表的修改权限。人员岗位变化或离职时，应及时复核并收回相应权限。审计日志用于记录关键访问和变更，但日志本身也要限制访问、防止篡改并设置合理留存周期。

数据库安全还需与传输加密、静态数据保护、备份恢复和应用层参数化查询配合。权限控制不能代替完整性约束，数据库备份也不能代替访问控制；这些措施保护的目标和生效环节不同。
