# 软件工程讲义

> 定位：事业单位计算机基础专业知识。重点掌握软件与软件工程、生命周期、开发模型、需求、设计、测试、维护和配置管理；不要求项目估算、关键路径、挣值或复杂建模计算。

## 一、软件与软件工程

### 1. 软件的定义和特点

软件是程序、数据和相关文档的完整集合。程序只是软件的一部分。

主要特点：

- 软件是逻辑产品，不是物理实体。
- 软件开发主要是设计和研制，不是传统制造复制。
- 软件不会像硬件那样物理磨损，但反复修改可能引入缺陷、使结构退化。
- 软件复杂度高，需求和运行环境会不断变化。
- 软件开发与维护依赖人员协作、方法、工具和管理。

### 2. 软件危机

软件危机是软件开发和维护中出现的一系列严重问题，表现为进度延期、成本超支、质量难保证、需求不满足、维护困难和文档不足。

主要原因包括软件规模和复杂度增长、需求不清或频繁变化、估计和管理不足、缺乏有效方法工具、人员沟通协作不良。

软件危机不是计算机硬件损坏，也不是某一个程序出现故障，而是软件开发和维护的系统性困难。

### 3. 软件工程

软件工程是把工程化、系统化、规范化和可量化的方法用于软件开发、运行和维护的学科。

三要素：

- 方法：完成分析、设计、测试等活动的技术。
- 工具：为方法提供自动化或半自动化支持。
- 过程：组织活动、角色、产品和质量控制。

目标是在成本和进度约束下获得满足需求、质量良好且易维护的软件。软件工程不只是编写代码，还包括需求、设计、测试、管理、文档和维护。

## 二、生命周期与过程模型

### 1. 生命周期

软件生命周期是软件从提出、开发、使用到退役的全过程。常见阶段：问题定义与可行性研究、需求分析、概要设计、详细设计、编码、测试、部署运行和维护。

阶段划分可以因组织和模型不同而变化。需求、测试、配置管理和质量保证常贯穿多个阶段，不能理解为每项活动只做一次。

### 2. 瀑布模型

特点是阶段明确、文档驱动、阶段成果经评审后进入下一阶段。优点是过程清晰、便于计划和审查；缺点是较晚看到完整产品，对需求变化适应较慢。

适合需求较稳定、技术成熟、强调文档和阶段审查的项目。经典瀑布强调顺序，但实际发现问题时仍可能反馈返工，并非物理上绝不能回退。

### 3. 原型模型

快速构建可演示或可运行的原型，让用户反馈，以澄清需求、验证界面或技术方案。

- 抛弃式原型：明确需求后废弃。
- 演化式原型：在工程控制下逐步发展为产品。

原型有助于解决需求模糊，但若直接把低质量原型当成最终系统，可能造成结构混乱和维护困难。

### 4. 增量、螺旋和敏捷

- 增量模型：把系统分批交付，每次提供可运行功能，能较早交付核心价值。
- 螺旋模型：迭代开发并突出风险分析，适合复杂、高风险项目，但管理成本较高。
- 敏捷开发：短周期迭代、持续反馈、响应变化、重视协作和可工作的软件。

敏捷不是不要计划、设计和文档，而是根据价值保留“够用且及时”的内容。Scrum 是常见敏捷框架，XP 强调测试驱动、结对编程、持续集成和重构等工程实践。

## 三、可行性与需求工程

### 1. 可行性研究

可行性研究用较小代价判断项目是否值得和能够实施。

- 技术可行性：现有技术、设备和人员能否实现。
- 经济可行性：成本与收益是否合理。
- 操作可行性：用户和组织是否能够接受、使用和维护。
- 法律可行性：是否符合法律、许可、合同和知识产权要求。
- 进度可行性：能否在规定时间内完成。

### 2. 需求分类

- 功能需求：系统必须做什么，如“用户能够查询信息”。
- 非功能需求：系统做得怎样，如性能、安全、可靠性、可用性、兼容性。
- 约束：必须遵守的平台、标准、语言、法规或接口限制。

“响应时间不超过 2 秒”是可度量的非功能需求；“系统应快速、友好”过于模糊，难以验证。

### 3. 需求获取与规格说明

常见获取方法：访谈、问卷、观察、会议/工作坊、文档分析和原型。不同方法应结合使用，并让用户确认。

需求规格说明书 SRS 描述系统必须做什么，是设计、开发、测试和验收的重要依据。良好需求应正确、完整、一致、明确、可验证、可追踪，并具有必要的优先级。

需求变更应先提出申请和影响分析，再审批、实施、验证并更新文档和追踪关系，不能口头修改后不留记录。

## 四、分析与设计

### 1. 结构化分析

数据流图 DFD 描述数据在系统中的流动和处理，基本元素是外部实体、加工、数据流和数据存储。DFD 说明“数据怎样流动和被处理”，不是程序控制流程图。

数据字典对数据项、数据结构、数据流和数据存储作统一说明。上下层 DFD 的输入输出应保持平衡。

### 2. 概要与详细设计

概要设计确定软件体系结构、模块划分、全局数据结构和模块接口，回答“系统由哪些部分组成”。

详细设计确定每个模块内部的算法、数据结构和处理过程，回答“模块内部怎样实现”。常用表示工具有程序流程图、N-S 图、PAD 图和伪代码。

程序流程图箭头表示控制流；DFD 箭头表示数据流，两者不能混同。

### 3. 模块化原则

高内聚、低耦合是模块设计的重要目标。

- 内聚描述模块内部各部分联系程度，功能越集中通常越好。
- 耦合描述模块之间依赖程度，接口越简单稳定、共享信息越少通常越好。

信息隐藏要求模块只公开必要接口，隐藏内部实现。降低耦合不能只理解为“传递数据越少越好”，还要避免全局共享、控制标志和直接访问其他模块内部。

### 4. 面向对象与 UML

- 封装：把数据和操作结合并隐藏实现。
- 继承：从已有类获得特征，表达 is-a 关系。
- 多态：同一接口可以有不同实现。

常见 UML 图：

- 用例图：描述参与者和系统功能。
- 类图：描述类、属性、操作和静态关系。
- 顺序图：按时间顺序描述对象消息。
- 活动图：描述流程、分支和并发。
- 状态图：描述对象状态及转换。
- 部署图：描述软件构件与硬件结点部署。

DFD 不是 UML 图，程序流程图也不等同 UML 活动图。

## 五、软件测试

### 1. 目的和原则

测试是为了发现缺陷而执行或检查软件的活动，不是为了证明软件绝对正确，也不可能通过有限测试找出所有错误。

测试用例至少包括输入、执行条件和预期结果。调试是在发现缺陷后定位原因并修改，测试与调试不是同一活动。

### 2. 黑盒与白盒

黑盒测试不考虑内部代码，根据需求和输入输出设计用例，又称功能测试。常见方法：

- 等价类划分：把输入划成有效和无效类别，每类选代表值。
- 边界值分析：测试边界及其邻近值。
- 判定表：处理多个条件组合和对应动作。
- 错误推测：根据经验选择容易出错情况。

白盒测试根据程序内部逻辑和结构设计用例，又称结构测试。常见覆盖包括语句覆盖、判定/分支覆盖、条件覆盖和路径覆盖。

条件覆盖和判定覆盖一般互不包含；覆盖率高也不能证明需求和业务逻辑完全正确。

### 3. 测试层次

1. 单元测试：测试函数、类或模块等最小单元。
2. 集成测试：测试模块接口和协作。
3. 系统测试：测试完整系统的功能、性能和安全等。
4. 验收测试：由用户或代表确认系统是否满足业务需求。

回归测试是在修改后重新执行相关测试，确认原有功能未被破坏。

自顶向下集成常使用桩模块模拟下层被调用模块；自底向上集成常使用驱动模块模拟上层调用者。

### 4. 静态与动态测试

静态测试不运行程序，包括评审、代码检查和静态分析；动态测试需要执行程序。验证关注“是否正确地构造产品”，确认关注“是否构造了正确的产品”。验收测试是重要的确认活动。

## 六、维护、质量与配置管理

### 1. 四类维护

- 改正性维护：修复已发现缺陷。
- 适应性维护：适应平台、法规、接口等环境变化。
- 完善性维护：增加功能、改善性能或可用性。
- 预防性维护：改善内部结构，降低未来故障和修改风险。

重构是在不改变软件外部可观察行为的前提下改善内部结构，通常属于预防性改进，需要测试保护。

### 2. 软件质量

常见质量属性包括正确性、可靠性、可用性、效率、可维护性、可移植性、安全性和可复用性。质量属性可能相互制约，例如更强安全控制可能增加使用复杂度，需要结合场景权衡。

文档记录需求、设计、接口、测试和使用维护信息，是沟通、评审、追踪和维护的重要依据，不是可有可无的附属品。

### 3. 配置与变更管理

软件配置管理包括配置项识别、版本控制、变更控制、状态记录、配置审计、构建和发布管理。

基线是经过正式评审和批准的配置项集合，之后只能按变更流程修改。版本控制支持历史记录和多人协作；分支用于并行开发，标签常标记发布点。

变更基本流程：申请 → 影响分析 → 审批 → 实施 → 测试验证 → 更新基线和文档。

## 七、软件项目与风险基础

项目管理关注范围、进度、成本、质量、人员、沟通和风险。基础考试只要求理解作用，不做 COCOMO、PERT、关键路径或挣值计算。

风险是尚未发生、可能影响目标的不确定事件。风险管理包括识别、分析排序、制定应对、监控。常见应对有规避、降低、转移和接受。已经发生并正在影响项目的是问题，不再只是风险。

增加人员不一定能任意缩短进度，新成员需要培训和沟通，任务也不一定能完全并行。

## 八、基础简答题答题框架

### 1. 定义表现原因类

先定义，再列主要表现，最后写原因和解决思路。软件危机、软件工程、风险均可使用。

### 2. 模型比较类

按“核心机制—优点—缺点—适用条件”比较。不能只写“某模型好”，要说明项目需求稳定性、风险和交付方式。

### 3. 方法区分类

按“是否查看内部结构—依据—常用方法—发现的主要问题”比较黑盒与白盒；按“对象—目标—顺序”比较四级测试。

### 4. 判断纠错

1. “程序等于软件”错误，软件还包括数据和文档。
2. “测试的目的是证明没有错误”错误，测试用于发现缺陷，有限测试不能证明绝对无错。
3. “敏捷开发不需要文档”错误，敏捷强调价值和及时性，不是取消文档。
4. “需求分析主要说明怎样编码实现”错误，需求分析重点说明系统必须做什么。
5. “集成测试主要检查单个模块内部算法”错误，集成测试重点检查接口和协作。
6. “软件不会磨损，因此不需要维护”错误，软件会因缺陷、需求和环境变化持续维护。
7. “DFD 箭头表示控制流”错误，DFD 表示数据流，程序流程图表示控制流。

## 九、考前检查

- 能说出软件定义、特点、软件危机表现和软件工程三要素。
- 能比较瀑布、原型、增量、螺旋和敏捷模型。
- 能区分功能、非功能和约束需求，说明 SRS 特点。
- 能区分概要/详细设计、DFD/流程图和常见 UML 图。
- 能比较黑盒/白盒、静态/动态和四级测试。
- 能说出四类维护、质量属性、配置管理和变更流程。
- 能按“定义—特点—作用—比较—判断纠错”完成基础简答。
