# 📚 软件工程

使用说明：🔴红色= 一级考点 / 核心术语 / 判断改错点（背下来）🔵蓝色= 简答题踩分关键词 / 开发模型特征 / 测试方法区分（考试时写上去就是分）⚫️黑色= 具体定义解释、例子、选择题常见正确/错误描述（理解即可）
第一章：软件工程概述
考点1：软件的定义与特点
🔴【软件的定义】：软件 =程序 + 数据 + 相关文档的完整集合。
🔴【软件的特点】：
软件是一种逻辑产品，不是物理实体。
软件的开发主要是研制，不是生产制造。
软件不会像硬件那样因物理磨损失效，但持续修改可能引入缺陷、破坏结构，使其出现“退化”；运行环境变化也会导致原本正确的软件不再适用。
软件的开发和运行受计算机系统限制。
⚫️【坑点】：
程序不等于软件（软件还包括数据和文档）。
软件既是逻辑实体，不是物理实体。
软件安装的越多，计算机性能不一定越好（过多软件会占用系统资源）。
“裸机”是指不装备任何软件的计算机。
考点2：软件危机（Software Crisis）
🔴【定义】：在软件开发和维护过程中遇到的一系列严重问题。
🔴【主要表现】：
进度滞后：软件开发进度难以预测。
成本超支：软件开发成本难以控制。
质量不可控：软件产品质量无法保证。
维护困难：软件产品难以维护。
文档缺乏：缺少适当的文档资料。
🔴【产生原因】：
用户需求描述不准确。
缺乏有力的组织和管理。
缺乏有力的方法学和工具支持。
软件规模越来越大，复杂度越来越高。
⚫️【坑点】：
软件危机不是“电脑坏了”或“硬件故障”，而是软件开发生产力跟不上硬件发展。
软件危机的表现不包括“软件依赖于硬件”（那是软件的特性之一）。
考点3：软件工程（Software Engineering）
🔴【定义】：采用工程的概念、原理、技术和方法来开发和维护软件，以经济地开发出高质量的软件并有效地维护它。
🔴【软件工程三要素】：方法（技术手段）+工具（自动化支撑）+过程（质量管理的纽带）。（口诀：方工过）
🔴【软件工程的目标】：在给定成本和进度的前提下，开发出具有适用性、有效性、可修改性、可靠性、可理解性、可维护性、可重用性、可移植性、可追踪性、可互操作性的软件产品。
⚫️【坑点】：软件工程的核心是管理复杂性，不是单纯地写代码。
第二章：软件生命周期
考点4：软件生命周期（SDLC）
🔴【定义】：软件从产生到报废的整个周期。
🔴【常见教材口径】：定义期（问题定义、可行性研究、需求分析）、开发期（概要设计、详细设计、编码、测试）、运行维护期。生命周期阶段的名称和划分会因过程标准与模型不同而变化，并非只能固定为“三期八阶段”。
⚫️【坑点】：
维护通常持续时间很长、累计成本占比较高，但具体比例随系统寿命、类型和统计口径而变，不能把60%～70%当作通用定律。
性能优化可能发生在设计、编码、测试和维护中，是软件工程活动的一部分；优化前应有可测量目标并避免牺牲正确性和可维护性。
详细设计属于软件开发期，不属于定义期或维护期。
第三章：软件开发模型（极高频）
考点5：瀑布模型（Waterfall Model）
🔴【核心特征】：阶段化、文档驱动，阶段成果经评审后进入后续阶段；经典瀑布强调顺序和基线控制，但实际可有反馈与返工，并非物理上绝对不能回退。
🔴【优点】：文档驱动，每个阶段都有明确的产出，易于管理和审查。
🔴【缺点】：需求变更极难，必须等全部做完才能看到效果。不适合需求不明确的项目。
🔴【适用场景】：需求较明确稳定、技术成熟、强调阶段审查和可追踪文档的项目。军工、银行等项目是否采用瀑布不能仅凭行业名称判断。
⚫️【坑点】：瀑布模型的最大优点是将软件开发各个阶段划分得十分清晰。
考点6：原型模型（Prototyping Model）
🔴【核心特征】：快速构建原型以澄清需求、验证方案并获取反馈。抛弃式原型在需求明确后废弃；演化式原型在工程控制下逐步演化为产品，不能混为“边看边改直到交付”。
🔴【优点】：能有效解决需求模糊的问题，降低风险，促进用户与开发人员的沟通。
🔴【缺点】：若不加控制，容易导致设计草率、结构混乱，原型可能被直接当作最终产品交付。
🔴【适用场景】：需求不明确、用户交互复杂的系统（如：UI界面设计、新概念验证）。
⚫️【坑点】：快速原型模型可以有效地适应用户需求的动态变化。
考点7：增量模型（Incremental Model）
🔴【核心特征】：分批次交付，每次交付一个可运行的子集（增量），逐步完善。
🔴【优点】：用户可以尽早拿到核心功能，分散风险，先交付核心功能。
🔴【适用场景】：大型项目，且核心功能需尽快上线的场景。
考点8：螺旋模型（Spiral Model）
🔴【核心特征】：迭代演化并由风险驱动，每轮通常经历目标确定、风险分析与化解、开发验证、下一轮计划。它综合了原型和系统化开发思想，突出风险分析。
🔴【优点】：强调风险规避，适合大型高风险项目。
🔴【缺点】：过分依赖风险评估经验，成本较高，对风险分析人员要求高。
🔴【适用场景】：内部开发的大型、高风险、复杂项目。
⚫️【坑点】：风险分析是螺旋模型的标志性活动，但其他过程也可以做风险管理，不能理解成“只有螺旋模型才允许分析风险”。
考点9：敏捷开发（Agile Development）
🔴【核心特征】：迭代增量交付、快速反馈、响应变化、重视人员协作和可工作的软件。敏捷强调“够用且有价值的文档”，不是不要文档。
🔴【代表性方法】：
极限编程（XP）：测试驱动开发、结对编程、持续集成。
Scrum：冲刺（Sprint）、每日站会、产品待办列表。
🔴【优点】：响应变化快，客户满意度高，交付迅速。
🔴【局限】：对协作、持续反馈和工程实践要求较高；规模扩大时还要处理跨团队协调、架构治理和合规文档。大型项目也可采用规模化敏捷或混合过程，不能笼统说“不适合”。
🔴【适用场景】：需求快速变化的互联网产品（如：APP开发、网站开发）。
⚫️【坑点】：敏捷开发强调**“拥抱变化”**，不是拒绝变化。

第四章：需求分析
考点11：需求分析的任务与产出
🔴【核心任务】：准确回答系统**“必须做什么”**（WHAT），而不是“怎样做”（HOW）。
🔴【主要工作】：
问题识别：明确功能需求、性能需求、环境需求、用户界面需求。
分析与综合：导出软件的逻辑模型。
编写文档：产出需求规格说明书（SRS）。
🔴【最终产物】：需求规格说明书（SRS）。
⚫️【坑点】：需求分析不仅确定功能，还要明确性能、安全、接口、数据、约束和验收标准。需求应正确、完整、一致、可验证、可追踪，并与利益相关者确认。
考点12：需求的分类（极高频判断/选择）
🔴【功能需求】：系统必须做什么（具体的业务功能）。
例：“用户输入账号密码，系统验证登录。”
例：“系统能够完成数据的增删改查。”
🔴【非功能需求】：系统做得好不好（质量属性、性能、安全、可靠性、可维护性）。
例：“系统支持500人并发访问。”
例：“页面响应时间小于3秒。”
例：“数据必须加密传输。”
🔴【约束需求】：开发过程中的限制条件（外部硬性规定）。
例：“必须使用MySQL数据库。”
例：“开发语言必须是Java。”
例：“必须在Windows环境下运行。”
⚫️【坑点】：
看到**“必须使用XX语言/平台/工具”** → 选约束需求，不是功能需求！
看到**“响应时间/吞吐量/并发数”** → 选非功能需求！
“系统需要支持1000个用户同时在线查询” →非功能需求（性能指标）。
考点13：需求获取方式（简答题）
🔵【常用方式】：
问卷调查法：向用户发放调查问卷收集需求。
会议讨论法：开发方和用户方召开需求讨论会议。
界面原型法：通过“界面原型”这一载体与用户沟通需求。
可运行原型系统法：在以往类似项目基础上修改得到可运行系统，让用户试用以挖掘需求。
访谈与现场观察：通过访谈利益相关者、观察真实业务流程识别显性与隐性需求。
文档分析与工作坊：分析现有规章、表单和系统资料，组织JAD/需求工作坊协调冲突并形成共识。
第五章：软件设计
考点14：模块设计原则（极高频）
🔴【核心原则】：“高内聚、低耦合”。
🔴【内聚（Cohesion）】：一个模块内部各元素之间结合的紧密程度。内聚越高越好（模块内部团结一致）。
内聚强度从低到高：偶然内聚 < 逻辑内聚 < 时间内聚 < 过程内聚 < 通信内聚 < 顺序内聚 <功能内聚（最强）。
🔴【耦合（Coupling）】：不同模块之间相互依赖的程度。耦合越低越好（模块之间独立性强）。
耦合强度从低到高：非直接耦合 <数据耦合< 标记耦合 < 控制耦合 < 公共耦合 < 内容耦合（最强）。
降低耦合不仅是“传得少”，还要让接口简单稳定、只传必要数据，避免共享全局数据、控制标志和直接访问另一模块内部实现。
⚫️【坑点】：
数据耦合、标记耦合、控制耦合、公共耦合的耦合性从低到高：数据 < 标记 < 控制 < 公共。
要减少模块间的耦合，应减少直接传递的信息量。
模块设计应追求低耦合、高内聚。
考点15：概要设计（总体设计）
🔴【主要任务】：
确定软件的体系结构（模块结构）。
模块划分。
确定全局数据结构。
定义模块之间的接口。
🔴【产出文档】：模块结构图和模块说明书。
考点16：详细设计
🔴【主要任务】：确定每个模块的算法和使用的数据结构。
🔴【常用工具】：程序流程图（PFD）、N-S图（盒图）、PAD图、伪代码。
🔴【程序流程图中的箭头】：代表控制流（不是数据流！）。
⚫️【坑点】：程序流程图中的箭头代表控制流，系统流程图中的箭头代表数据流。
第六章：软件测试（极高频，题型多样）
考点17：测试的目的
🔴【测试目的】：发现软件中的错误，不是为了证明软件正确。
🔴【经典名言】：测试是为了发现错误而执行程序的过程。好的测试用例是能发现迄今为止尚未发现的错误的用例。
⚫️【坑点】：软件测试的目的是发现软件的错误，不是“找出软件的所有错误”或“证明软件是正确的”（后者不可能实现）。
考点18：黑盒测试（Black-box Testing）
🔴【定义】：不看内部代码，只关注输入和输出（把程序当黑盒子）。又称功能测试。
🔴【常用方法】：
等价类划分：将输入数据划分为若干有效等价类和无效等价类，从每类中选一个代表测试。
边界值分析：重点测试边界及其邻近值（如整数范围1～100可测0、1、2、99、100、101）。它对边界缺陷很有效，但不存在对所有缺陷都“能力最强”的单一黑盒方法。
错误猜测法：根据经验猜测可能出错的情况。
因果图：描述多种条件组合的测试用例。
⚫️【坑点】：
黑盒测试的测试用例不是根据程序内部逻辑设计的（那是白盒测试）。
边界值分析尤其擅长发现边界处理错误，应与等价类、判定表、状态迁移等方法组合，不能绝对称为任何场景都最强。
等价类划分法不是白盒测试技术，是黑盒测试技术。
常用的黑盒测试方法包括：等价类划分、边界值分析、错误猜测、因果图。（α测试和β测试是测试阶段，不是测试方法）
考点19：白盒测试（White-box Testing）
🔴【定义】：看内部逻辑结构，检查代码路径是否走通。又称结构测试。
🔴【常见覆盖标准】：
语句覆盖（最弱）：每条可执行语句至少执行一次。
判定覆盖（分支覆盖）：每个判断的真/假分支至少执行一次。
条件覆盖：每个判断中每个条件的真/假至少出现一次。
判定/条件覆盖：同时满足判定覆盖和条件覆盖。
条件组合覆盖：每个判断中所有条件的组合至少出现一次。
路径覆盖（最强）：程序中所有可能的执行路径都至少走一遍。
⚫️【坑点】：
判定覆盖通常包含语句覆盖；条件覆盖与判定覆盖一般互不蕴含，所以不能简单写成“语句<判定<条件<路径”的全序。判定/条件覆盖同时满足二者，条件组合覆盖更强。
条件覆盖不一定包含判定覆盖！（极高频坑点：两个独立条件都出现了，但整体判定可能没走全）。
有限、无环控制流中覆盖所有可行路径是很强的标准；存在循环时路径可能无限，实际通常覆盖选定的独立路径或限定循环次数。结构覆盖率高也不能证明需求和业务逻辑正确。
考点20：测试阶段（顺序必记）
🔴【测试顺序】：单元测试 → 集成测试 → 系统测试 → 验收测试。
🔴【单元测试】：测试最小模块（如函数、类、方法）。用例主要依据详细设计设计。
🔴【集成测试（组装测试）】：逐步组合模块，重点检查接口、数据传递、调用关系和协作行为；可发现设计、实现及环境配置等多类缺陷，不能只说发现设计阶段错误。
🔴【系统测试】：测试整个系统的功能和性能是否满足需求。
🔴【验收测试】：用户确认系统是否满足需求。
Alpha测试：通常在开发组织可控环境中，由内部人员或受邀用户测试，开发方可观察和支持。
Beta测试：发布给有限外部用户在真实或接近真实环境中试用并反馈。两者定义会因组织发布流程略有差异。
⚫️【坑点】：
集成测试测试的是模块接口，不是内部功能。
单元测试的用例依据详细设计设计，不是需求分析。
集合测试（集成测试）的主要目的是发现设计阶段的错误。
第七章：软件维护
考点21：软件维护的类型（简答题必背）
🔴【维护特点】：维护通常贯穿软件交付后的较长时期，累计投入常很高；是否“花费最多”取决于系统寿命与统计口径。
🔵【四大维护类型】：
改正性维护：识别和纠正已发现的错误（修复缺陷）。
适应性维护：使软件适应外部环境变化（平台、法规、接口等）。
完善性维护：根据新需求扩充功能、改善性能或可用性。
预防性维护：改善内部结构、可维护性或可靠性，以降低未来故障和修改风险。
⚫️【坑点】：
许多经典统计中完善性维护占比较大，但比例随项目和口径显著变化，题目未给教材数据时不应死记20/25/50/5。
改正性维护是修复Bug，适应性维护是适应外部环境变化。
第八章：软件项目管理
考点21A：验证、确认与评审（高频辨析）
🔴【Verification，验证】：检查工作产品是否符合上一阶段规定和规格，常概括为“是否正确地构造产品”。评审、检查、静态分析不要求运行程序，也属于质量保证手段。
🔴【Validation，确认】：检查最终系统是否满足用户真实需要和预期用途，常概括为“是否构造了正确的产品”。验收测试是重要确认活动。
⚫️【坑点】：软件测试通常需要运行被测程序，静态评审不运行程序；二者都能发现缺陷，不能把“测试”与“评审”混为一种方法。
考点21B：UML常用图（岗位考试高频）
🔴【用例图】：描述参与者与系统功能需求的关系，常用于需求分析；参与者是角色，不一定等同某个具体人。
🔴【类图】：描述类、属性、操作及关联、聚合、组合、泛化等静态关系。
🔴【顺序图】：按时间顺序描述对象之间的消息交互；【活动图】描述业务流程、控制流和并发分支；【状态机图】描述对象状态及事件触发的迁移。
⚫️【坑点】：数据流图DFD是结构化分析工具，不是UML图；程序流程图描述算法控制流，也不等同于UML活动图。
考点22：可行性研究
🔴【目的】：用最小的代价在尽可能短的时间内确定问题是否能够解决。
🔴【四种可行性】：
经济可行性：成本-效益分析。
技术可行性：从技术角度研究实现系统的可行性（对系统所要求的功能、性能及限制条件进行分析）。
操作可行性：系统的运行方式在用户组织中是否可行。
法律可行性：是否触犯法律或有侵权行为。
考点23：软件质量与文档
🔴【软件文档的作用】：不仅是开发各阶段的依据，而且影响软件的可理解性。
🔴【软件配置管理】：
“检出（Check-out）”和“检入（Check-in）”用于受控获取、提交配置项版本，可辅助实现存取控制、版本管理和并行修改协调；现代分布式版本控制并不一定采用独占检出模式。
🔥 本章高频简答题汇总
1. 什么是软件危机？主要表现有哪些？
🔵【踩分点】：软件危机是在软件开发和维护中遇到的一系列问题。主要表现：进度滞后、成本超支、质量不可控、维护困难、文档缺乏。
2. 简述软件生命周期的三大时期和主要阶段。
🔵【踩分点】：
软件定义期：问题定义、可行性研究、需求分析。
软件开发期：概要设计、详细设计、编码、测试。
软件运行期：维护（花费最多、时间最长）。
3. 比较瀑布模型和原型模型。
🔵【踩分点】：
瀑布：线性顺序、需求明确时适用、文档驱动、变更困难。
原型：边看边改、需求模糊时适用、快速构建、容易导致结构混乱。
4. 简述螺旋模型的核心特点。
🔵【踩分点】：螺旋模型 =瀑布模型 + 原型模型 + 风险分析。每轮迭代都包含风险分析是其独有特征，适合大型高风险项目。
5. 区分功能需求、非功能需求和约束需求。
🔵【踩分点】：
功能需求：系统必须做什么（具体功能）。
非功能需求：系统做得好不好（性能、安全、可靠性等）。
约束需求：开发的外部限制条件（必须使用XX语言/平台）。
6. 简述黑盒测试和白盒测试的区别。
🔵【踩分点】：
黑盒测试：不看内部代码，只测输入输出（功能测试）。方法：等价类划分、边界值分析。
白盒测试：看内部逻辑结构（结构测试）。方法：语句覆盖、判定覆盖、条件覆盖、路径覆盖。
7. 简述软件测试的四个阶段及其任务。
🔵【踩分点】：
单元测试：测试每个模块内部。
集成测试：测试模块之间的接口。
系统测试：测试整个系统。
验收测试：用户确认是否满足需求（Alpha = 开发环境，Beta = 真实环境）。
8. 简述软件维护的四种类型。
🔵【踩分点】：
改正性维护：修复已发现缺陷。
适应性维护：适应平台、法规、接口等环境变化。
完善性维护：扩充功能、改善性能或可用性。
预防性维护：改善内部质量，降低未来故障和修改风险。
